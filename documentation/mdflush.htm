LinuxKI Warning

 

High kworker CPU usage when using software RAID (md driver) with barrier writes

05/10/2018

 

Problem

 

Linux server experienced severe performance degradation after upgrading from RHEL 6.8 to RHEL 7.4 with extremely high System CPU usage.    Thousands of kworker daemon tasks were observed to be active on the system.   

 

Background

 

When using Linux Software RAID, also known as Multiple Device (md) driver, hundreds or even thousands of kworker daemon tasks will be active.    The kworker daemon tasks may only take a small amount of CPU time each, but cumulatively, it can be dominating the CPU usage.    

 

There have been several changes to Linux over the past few years which contributes to the issue:

  • The Multiple Device (md) driver was re-writen in Linux version 2.6.37   
  • Pool workqueues were introduced in Linux version 3.9
  • The Asynchronous IO completions started using workqueues in Linux version 3.12.

The issue is caused by performing a lot of asynchronous IO activity with barrier writes.   In this case, the system was using Oracle on ext4 filesystems, and the Oracle DB writer processes periodically flush the dirty data from the SGA cache asynchronously to disk.      While the customer was using a disk array with a non-volatile cache, the disk array advertised as "write cache enabled".    

 

kernel: sd 3:0:0:6: [sdbc] Write cache: enabled, read cache: enabled, doesn't support DPO or FUA

kernel: sd 3:0:1:6: [sded] Write cache: enabled, read cache: enabled, doesn't support DPO or FUA

...

 

As a result, each asynchronous IO would need to go through the barrier write flush code and only a single thread could go through the code at one time, and the rest of the kworker tasks would go to sleep in md_flush_request().     As the writes started to backup, more kworker daemons were created to handle the additional workload.    But this made matters worse, as when the kworker threads were woken up, ALL of the sleeping kworker threads were woken up at one, but only one of them could perform the barrier write flush.    This behavior is commonly known as a "thundering herd".

 

Solution

 

The issue can be solved by simply mounting the ext4 filesystem with the nobarrier or barrier=0 options.

 

​LinuxKI analysis

 

Just because there are busy kworker tasks does not mean the problem is related to the md driver and barrier writes.     You need to verify what the kworker tasks are doing.
