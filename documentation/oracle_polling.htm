LinuxKI Warning

 

​​​​Excessive poll() calls by Oracle

11/28/2018

 

Background

 

The poll() calls by Oracle server processes can be used to detect in-band breaks from clients that do not support out-of-band breaks. A break is an exception such as a CNTL-C to abort a transaction. Data exceptions can be transmitted as part of the regular data stream (in-band) or as a separate asynchronous message (out-of-band). By default, Oracle uses out-of-band (OOB) breaks. OOB breaks are handled more quickly and with less overhead as they interrupt the flow of data and Oracle does not need to poll for them. There is a sqlnet.ora parameter (DISABLE_OOB=on) than can be used to disable OOB for all clients.

 

Generally it is best to use OOB (DISABLE_OOB=off) and most Oracle servers are set up that way. However, some clients do not support OOB and so for those clients the server must use in-band breaks instead. As mentioned above, in-band breaks are part of the data stream and Oracle must poll for them, which it does rather aggressively.  This can cause excessive polling by the Oracle server processes, especially CPU intensive processes. 

 

These poll() calls are typically very fast, so it doesnt usually cause any major issues, only a slight performance impact.

While the system calls are quite fast, it is a break in the normal code execution to repeatedly check the socket descriptor of the client, and can have a small impact on performance.

 

With LinuxKI version 5.8, the Kparse report will produce a warning if excessive poll() calls are observed by an Oracle server process if it shows up in the following section:

NOTE: Do not confuse these fast poll() calls with poll() calls that may sleep and have a higher Average (Ave) time, which typically indicates that the Oracle server process is on a RAC wait.

 

Solution

 

For the most part, the overhead of the excessive poll() calls is quite small and can often be ignored. However, if you are trying to maximize throughput for a benchmark or other reads, the poll() overhead can be drastically reduced by tuning the BREAK_POLL_SKIP parameter in the sqlnet.ora file on both the clients and servers.

 

SQLNET.ORA: BREAK_POLL_SKIP

 

A lesser known SQL*Net parameter. This value specifies the number

of packets to skip before checking for a user break.

Rules of thumb here:   

- The higher the value, the less frequent CTRL-C checking, and the     

  less CPU overhead used.   

- The lower the value, the more frequent CTRL-C checking, and the    

  more CPU overhead used.   

The default for this parameter is 4.​

 

A good value for BREAK_POLL_SKIP to reduce the amount of polling but still allow in-band breaks is 10000.
