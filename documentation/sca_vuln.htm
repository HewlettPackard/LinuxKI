LinuxKI Warning

 

Side-Channel Attack (Spectre/Meltdown) mitigation can negatively impact performance

Date: 01/16/2019

Problem

In early 2018, various hardware chip manufacturers including Intel, AMD, IBM, and ARM, announced several security vulnerabilities known as Side-Channel Attacks.   In order to address these security vulnerabilities, a combination of firmware and Linux operating system changes were introduced.    These changes can have minor to severe implications on system and application performance.  

Identifying if Side Channel Attack (SCA) fixes are present

Applications, software, and data are often migrated to new servers, or the Linux OS of an existing server can be upgraded.    Performance on new faster hardware or new Linux OS may be less than expected or may actually decline after the migration as the old environment may not have the Side-Channel Attack mitigations, while the new hardware with a later Linux OS may have the Side-Channel Attack mitigations.

To determine if the SCA are mitigated, the following can be performed:

$ for file in `ls /sys/devices/system/cpu/vulnerabilities/* 2>/dev/null`; do \

    echo $file `cat $file`; done

/sys/devices/system/cpu/vulnerabilities/l1tf Mitigation: PTE Inversion

/sys/devices/system/cpu/vulnerabilities/meltdown Mitigation: PTI

/sys/devices/system/cpu/vulnerabilities/spec_store_bypass Vulnerable

/sys/devices/system/cpu/vulnerabilities/spectre_v1 Mitigation: __user pointer sanitization

/sys/devices/system/cpu/vulnerabilities/spectre_v2 Mitigation: Full generic retpoline

From the above, 4 of the 5 vulnerabilities have been mitigated (fixed).    As a result, there will be some performance impact.   However, the extent of the performance impact may be minor or severe, depending on the applications being executed.

LinuxKI and Side-Channel Attack (SCA) mitigations

The Side-Channel Attack mitigations leave very little evidence in the LinuxKI data.   The only real difference in the data which can be seen is an increase in the Cycles per Instruction (CPI).    The CPI metric can be gathered with LinuxKI by enabling the Advance CPU statistics.   For example:

$ runki -R

The following LinuxKI data shows the CPU metrics from a Gen8 server running RHEL 6.9 (2.6.32-696.3.2):

After Migrating to Synergy with Skylake CPUs running RHEL 7.4 (3.10.0-693.21.1)

 

node  ncpu     Total Busy          sys          usr         idle      LLC_hit%    CPI   Avg_MHz

   0 [ 16] :       34.27%        5.91%       28.24%       65.51%        72.91%   1.51   3700.00

   1 [ 16] :       28.96%        5.11%       23.81%       71.00%        66.28%   1.46   3700.00

Total              31.93%        5.60%       26.25%       67.94%        70.06%   1.49   3700.00

 

You can see the CPI increased from 1.31 to 1.49.    The CPI metric can change throughout the day and can be caused by a variety of reasons unrelated to the SCA mitigations (such as memory locality and program optimizations).    The performance increased from the old environment by 5-10%, but with newer hardware and CPU frequency that is 20% higher, the performance increase was less than expected.    

After disabling the Side-Channel Attack mitigations and re-running the workload, the CPI dropped by ~28%, from 1.49 to 1.08, and the overall performance of the new Synergy server was 35% better than the Gen8 server.    In this case, the Side-Channel Attack mitigations had a 25-30% impact on performance and transaction response times.

Unfortunately, you cannot look at a single environment and conclude that the CPI is high or low.   You would need to compare 2 environments/servers running the same workload in order to compare the CPI.

Summary

Be aware that the mitigations for the Side-Channel Attacks (Spectre/Meltdown) can have performance impacts, and these impact can be minor of severe.    The SCA mitigations should be checked before and after any server migration or Linux OS upgrades.
